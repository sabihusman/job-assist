# Application Audit — June 2026

Multi-agent adversarial audit of the full application (2026-06-10/11).
Method: 11 parallel finder agents swept the codebase by subsystem; every
finding was then independently challenged by 2 adversarial verifiers
(correctness + impact lenses). CONFIRMED = all verifiers upheld the claim;
POSSIBLE = split verdict; KILLED = all verifiers refuted it.

Alongside this review, six deterministic layers ran green: ruff, mypy,
pytest (full DB suite in Docker), biome, tsc, vitest, plus the OpenAPI
snapshot gate and a live prod smoke pass.

## HIGH findings

### [CONFIRMED] Changed posting (retitle/location change) creates a JobPosting permanently orphaned from its PostingSource — no apply link, corrupted provenance, temporary duplicate

**File:** `apps/api/src/job_assist/services/ingestion.py`

JobPosting is deduped by content_hash = sha256(company+normalized_title+locations), but PostingSource is upserted by (ats, source_job_id). When an ATS edits a live posting's title or location list (common: adding a remote option changes locations_normalized), the next ingest computes a NEW content_hash, so _upsert_job_posting inserts a brand-new JobPosting. _upsert_posting_source then finds the EXISTING source row by (ats, source_job_id) and only refreshes raw_payload + fetched_at — it never re-points job_posting_id to the new posting. Consequences: (1) the new JobPosting never gets a PostingSource, so the operator-facing list (main.py:2731-2737 recent_ps lateral on PostingSource.source_url) shows it with NO source/apply URL forever, and the ats filter (postings_query.py:230-237, EXISTS on PostingSource) can never match it; (2) the old source row's raw_payload now describes the NEW posting while pointing at the OLD JobPosting (provenance corruption); (3) the old JobPosting stays open and visible as a duplicate for up to 7 days until mark_stale_postings closes it. This directly answers the hunt question 'could a changed posting be missed or duplicated?' — it is duplicated, and the surviving copy is crippled.

### [CONFIRMED] ContactEditForm trims the input on every keystroke — typed spaces are eaten, multi-word values get saved concatenated

**File:** `apps/web/src/components/contacts/ContactEditForm.tsx`

Every text field in the contact edit form is a controlled input whose onChange immediately passes e.target.value through emptyToNull(), which calls .trim() (lines 54-56). When the operator types a trailing space, the trimmed value is written to state and React resets the input to the trimmed string, so the space never survives. Typing 'Vice President' sequentially yields 'VicePresident'; the same applies to current_employer, notes (a whole textarea — sentences become run-together words), preferred name, etc. The mangled value is then PATCHed to the API and persisted. Trim should happen on blur/submit (diffPatch time), not per keystroke. Affects lines 101, 112, 124, 134, 146, 154, 163.

### [CONFIRMED] Daily mark-stale closes warm-path postings whenever one weekly sweep touch is missed — and the health dot stays green while it happens

**File:** `apps/api/src/job_assist/services/ingestion.py`

mark_stale_postings closes ANY open posting with last_seen_at older than 7 days, with no awareness of sweep cadence (no source/cohort filter). It runs daily at ~06:05+ UTC from ingest-daily.yml (the 'Mark stale postings' step, if: always()). Warm-path postings get last_seen_at refreshed ONLY by the weekly Sunday 13:00 UTC sweep (warm-path-ingest.yml cron '0 13 * * 0'), so their age at the Monday mark-stale run is 7d17h if Sunday's touch did not land. The tolerance is therefore ~17 hours per WEEK, versus the documented intent that 'A 7-day floor tolerates a board being unreachable for a few consecutive days' (ingestion.py:463-465) — that guarantee only holds for daily-swept boards. Triggers: (a) a single per-employer Apify failure during the Sunday sweep (fantastic_ingest.py docstring: 'swallows per-employer failures into a failed run') leaves that employer's postings unrefreshed → ALL of them are wrongly closed Monday ~06:05 and vanish from Triage/strategy views until the NEXT successful Sunday sweep reopens them (up to 6 days); (b) a skipped/failed weekly workflow run (GitHub drops schedules under load and disables them after repo inactivity) closes the entire warm-path corpus Monday morning. Worse, fantastic_ingest.py:124-129 stamps TargetCompany.last_swept_at EVEN ON A FAILED RUN, and the health check reads MAX(last_swept_at) over the cohort (main.py:4087-4098) — so warm_path_fresh stays green for case (a); the only alert is a generic 'N failed ingest_run(s)' red for 26h that never says postings were closed, and for case (b) warm_path_fresh only trips ~2 days AFTER the closures (9-day window vs 7-day mark-stale floor). Live postings shown as closed is wrong results reaching the operator.

### [CONFIRMED] DetailPanel's ReasonPicker does not pause useTriageKeyboard — keys 1-4 double-dispatch, firing two contradictory mutations on the same posting

**File:** `apps/web/src/app/page.tsx`

The page keyboard hook is only disabled for the card-level picker: enabled = !recordAction.isPending && reasonPickerCardId === null (page.tsx:310). DetailPanel's picker uses its own local state (DetailPanel.tsx:134 reasonOpen, mounted at lines 316-318) that the page never sees. Both listeners are window keydown listeners, so with the panel's picker open (operator clicked Pass in the detail action bar), pressing '1' fires BOTH useTriageKeyboard.onAction1 -> recordAction.mutate({action_type:'interested'}) AND ReasonPicker.onSelect('wrong_role') -> recordAction.mutate({action_type:'not_interested', reason:'wrong_role'}) on the same selectedId — two racing POSTs to /postings/{id}/state, last-write-wins, final posting state nondeterministic, plus a spurious action-history row either way. '3'/'4' similarly commit applied/snoozed on top of the chosen reason; '2' opens the card-level picker AND commits a wrong_location pass simultaneously; Escape closes the whole panel instead of just the picker. This is exactly the keystroke fight the useTriageKeyboard docstring (lines 14-19) says the enabled flag exists to prevent — it's wired for the TriageCard picker but not the DetailPanel one.

### [CONFIRMED] Poll watermark is built from the sender-controlled Date header; one future-dated email permanently freezes the poll

**File:** `apps/api/src/job_assist/gmail/backfill.py`

run_poll derives its watermark from MAX(outcome_event.received_at), and received_at comes from the RFC-2822 Date header (client.py parse_message, lines 129-135), which is set by the sender's clock — while Gmail's `after:` operator filters on internalDate (Gmail's receipt time). Two consequences: (1) any sender whose clock is ahead inflates the watermark and the next poll's `after:` query silently skips every message Gmail receives inside the skew window; (2) a single message with a far-future Date header — classic for spam — pushes MAX(received_at) into the future and the poll is dead forever: `after:<future epoch>` matches nothing, so no rows insert, so the watermark never moves. Crucially, the orchestrator inserts an outcome_event for EVERY classified message including `unrelated` (backfill.py lines 271-303 — _is_job_related only gates company resolution, not the insert), so spam/newsletters feed the watermark. The failure is invisible: record_sweep marks each empty poll 'success' and gmail_healthy stays green. Recovery requires the operator to notice missing outcomes and manually run the date-range backfill.

### [CONFIRMED] Second 'Load more' click silently drops a 100-row window from the triage list

**File:** `apps/web/src/app/page.tsx`

There is exactly one extra-page hook (page.tsx:83) and items concatenates page1 + the CURRENT extra page only (page.tsx:105-106). The Load More button stays rendered while hasMore (line 361) and each click does setExtraOffset(items.length) (line 364). First click: extraOffset=100, items = rows 0-199. Second click: setExtraOffset(200) re-keys the single extra query to offset=200, so its data becomes rows 200-299 and the previously loaded rows 100-199 vanish from the rendered list — items = rows 0-99 + 200-299, 'showing 200 of N' with a silent 100-row hole the operator never triages. With ~716 pending postings (DEFAULT_FILTERS comment, filters.ts:36-39), reaching the full queue requires multiple clicks, so this is the normal path, not an edge case — the PR #70 docstring ('Load More fetches the next page') only ever worked for one click. Related offset-math hazard in the same line: items.length is computed after optimistic removals, so clicking Load More while a mutation is settling sets an offset smaller than page1's true server window, duplicating rows (duplicate React keys on posting.id in TriageList).

### [CONFIRMED] Unsaved edits leak across contacts: ContactEditForm state survives a contact switch and can be saved onto the wrong contact

**File:** `apps/web/src/components/contacts/ContactDetailPanel.tsx`

ContactDetailPanel renders <ContactEditForm contact={data} /> with no key={data.id}, and ContactEditForm holds its edits in useState({}) (ContactEditForm.tsx:59) with no reset effect when the contact prop changes. The panel's useEffect (lines 68-71) resets only the archive/unarchive mutations. If the operator edits contact A without saving, then clicks contact B whose detail is already in the React Query cache (any previously-opened contact within gcTime), useContactDetail returns B's data immediately — the loading-skeleton branch never renders, the form never unmounts, and A's pending edits remain in state. The form then displays A's edited values over B's fields, and diffPatch(B, edits) makes Save PATCH contact B with contact A's values (e.g. A's email/employer written onto B). Only an uncached switch (which unmounts the form via the skeleton) avoids it.

### [POSSIBLE (split verdict)] apply_hard_rules crashes on ingest-time plain-str enum fields, silently losing the verdict (role_filter and seniority_levels rules)

**File:** `apps/api/src/job_assist/triage/hard_rules.py`

At ingest, new JobPosting rows carry plain-str role_family/seniority_level in memory: NormalizedPosting declares `role_family: str = "other"` and `seniority_level: str = "unknown"` (apps/api/src/job_assist/adapters/base.py:82-83), `_create_job_posting` assigns them verbatim (services/ingestion.py:296-297), and `_eval_hard_rules` runs on that in-memory object before commit (ingestion.py:138; SAEnum does not coerce on attribute set, only at bind). Rule 2 (line 197) does `posting.role_family.value` — because StrEnum members hash/compare equal to their value strings, the plain str passes the `in _PM_FAMILIES` membership test and then `.value` raises AttributeError. Rule 8 (line 260) does `posting.seniority_level.value` whenever cfg.seniority_levels_included is set — so EVERY new posting surviving rules 1-7 raises. The exception is swallowed by `_eval_hard_rules` (ingestion.py:404-420, 'log + swallow'), so hard_rule_failed stays NULL — which /postings reads as PASSED — and hard_rules_evaluated_at is never stamped. Net effect: with the seniority filter enabled (a PR #43 feature), out-of-band postings reach the operator queue as if they passed, and PM postings at non_pm_only companies are never marked. This contradicts score_seniority's own docstring claim that 'the hard-rule filter (PR #43) already drops out-of-band seniority at ingest time' (scoring.py:197-199), and is the exact str-vs-enum failure mode `_period_label` (hard_rules.py:115-121) documents and fixed for salary_period — but missed for these two fields. DB-loaded rows (re-eval endpoint, update path) get real enum members and work, which is why the bug only manifests on first-sight ingest.

## MEDIUM findings

### [CONFIRMED] Alumni badge click-through can land on an empty contacts list when display_name is outcome-derived

**File:** `apps/web/src/components/shared/RepeatSignalBadges.tsx`

The 'N alumni here' badge links to /contacts?company=<sig.display_name ?? companyName>. display_name is chosen server-side as the most-common raw name across the UNION of outcome-derived names and contact employer strings (company_signals.py lines 128, 162, 180), so when outcome rows dominate it is typically the linked TargetCompany.name or a subject extraction — not the contact's current_employer text. The contacts list endpoint filters with a substring match `Contact.current_employer.ilike(f"%{employer}%")` (apps/api/src/job_assist/main.py:1091), whose comment explicitly says the design goal is 'an over-match beats a silent under-match'. But the under-match happens in the other direction: if 6 outcome rows say 'Stripe, Inc.' (linked name) and the one alumni contact's employer is 'Stripe', the badge links to ?company=Stripe%2C+Inc. and ILIKE '%Stripe, Inc.%' matches no contact — the operator clicks a badge promising '1 alum here' and gets an empty filtered list. The link should be built from a contact-derived employer string (or the filter should normalize), not the union-wide most-common name.

### [CONFIRMED] Classifier violates its 'never raises for malformed model output' contract: valid non-object JSON (array/string) raises AttributeError

**File:** `apps/api/src/job_assist/gmail/classifier.py`

classify()'s docstring (lines 190-192) promises 'Never raises for malformed model output — returns unclassified'. The non-JSON path is handled, but when Gemini returns syntactically valid JSON that is not an object — a top-level array like `[{...}]` or a bare string, both of which happen with response_mime_type=application/json — json.loads succeeds and the result goes straight into _coerce_result, whose first statement is `payload.get(...)` (line 101). On a list/str that raises AttributeError, which is NOT caught: the surrounding try only catches pydantic ValidationError (lines 232-239) — a dead handler, since _coerce_result clamps every field and cannot realistically raise ValidationError. The AttributeError propagates to the orchestrator, which counts it as classifier_errors and skips the message with no row written (backfill.py:261-269). Net effect: a wasted paid Gemini call, and the message becomes a candidate for the permanent watermark drop in finding #2. The same gap exists for relink_unmatched's classifier path (outcome_relink.py:176-183).

### [CONFIRMED] Clearing looking_for_text leaves stale similarity_score corpus-wide and the semantic blend keeps feeding fit_score from a deleted profile vector

**File:** `apps/api/src/job_assist/services/embeddings.py`

When the operator clears looking_for_text, embed_profile_if_changed drops the profile vector (lines 408-415) and returns False. The PUT /operator/profile hook only runs recalibrate_similarity + rescore_open_postings when the return is True (main.py:2025-2036 `if changed:`), and recalibrate_similarity itself no-ops when the profile isn't embedded (embeddings.py:662-663) and never NULLs similarity_score. So every posting keeps the calibrated similarity_score computed against the now-deleted vector, and score_posting keeps blending it at 20% weight indefinitely — score_semantic_fit's docstring promises 'Returns None when the row isn't calibrated yet (posting or profile not embedded)' (scoring.py:427-430), but the column stays non-NULL so the ghost signal keeps reaching fit_score and the Best-fit sort. Meanwhile nearest_postings correctly reports available=False, so the operator gets contradictory states. Fix shape: on the clear path, also NULL similarity_score (and rescore) or have recalibrate clear scores when the profile vector is absent.

### [CONFIRMED] Concurrent sweeps (cron poll overlapping a 5-10 min backfill or a slow poll) double-spend Gemini calls, then IntegrityError on the unique email_message_id aborts the batch and fails the sweep

**File:** `apps/api/src/job_assist/gmail/backfill.py`

The idempotency check is a one-shot snapshot: _run_email_ingest pre-loads already-classified message ids ONCE before the loop (lines 222-234) and never re-checks. There is no locking on the gmail endpoints (sweep_claim.py exists but is only used for ingestion sweeps). The backfill endpoint is documented as taking 5-10 minutes under the 4s/request Gemini throttle (main.py:1729, classifier.py:42), and the poll is fired by an external cron (main.py:1771 'every 15 minutes') — so an operator-run backfill realistically overlaps a cron poll over the same recent messages. Both runners snapshot already_seen before the other inserts, both pay a Gemini call per shared message (wasted paid/throttled API spend), and the slower runner's next batch commit (every 25 inserts, lines 306-307, or the final commit at 312) hits the uq_outcome_event_email_message_id constraint: the IntegrityError rolls back up to 24 other pending, legitimately-new outcome rows along with the duplicate, propagates out, and record_sweep marks the whole sweep 'failed'. The constraint correctly prevents duplicate rows, but the docstring's claim that re-running 'is safe' does not hold concurrently, and the rolled-back messages' Gemini classifications are pure waste (and, via finding #2, those messages can then fall behind the watermark if the other runner advanced it).

### [CONFIRMED] Daily reclassify cron permanently re-buys Gemini calls for rows the LLM already classified as 'other'/'unknown'

**File:** `apps/api/src/job_assist/main.py`

The only_unclassified selection (role_family='other' OR seniority='unknown', closed_at IS NULL) has no exclusion for rows the LLM sweep has ALREADY processed (no classified_at IS NULL or classifier_version filter). A posting the LLM itself judges 'other' stays in the bucket forever, gets re-selected on a later sweep (ordering is classified_at ASC, so once the never-classified backlog drains it cycles the oldest re-classified rows), and is sent to Gemini again — producing the same answer with the same CLASSIFIER_VERSION, plus a rescoring pass. With the cron firing daily at 07:30 (limit 50), a steady-state population of N open LLM-confirmed-'other' rows costs up to min(N,50) wasted paid generate_content calls every day, indefinitely. This contradicts the workflow's documented intent: 'This cron sweeps those regex-failures daily so the inflow stays classified' (enrich-classifier.yml:12) — these rows are not inflow and re-running the same model version on them cannot change the result (the endpoint itself documents the idempotence: 're-running with the same LLM response produces the same values').

### [POSSIBLE (split verdict)] Latest-wins fold skips unattributable events, so a generic-subject rejection never flips a thread out of 'alive'

**File:** `apps/api/src/job_assist/services/company_signals.py`

The docstring (lines 80-83) promises per-thread latest-wins: 'its stage is its LATEST event's stage (latest-wins, so a rejection after a confirmation flips the thread out of "alive")'. But attribution happens per-EVENT before the fold: any row that is unlinked AND has a generic subject (company_from_subject returns None — its own docstring cites "Update on Your Application" as the canonical generic subject) is skipped at lines 121-123 and never reaches the latest dict. So in a thread whose confirmation WAS attributable ('Thank you for applying to Acme') but whose later rejection has a generic subject, latest[("acme", thread)] stays at the confirmation and the thread is counted in active_apps forever — even though the thread_id ties the rejection to the company. The same per-(key, thread) bucketing also double-counts one thread under two keys when its events attribute to different non-subset names (e.g. linked TargetCompany 'Meta' vs subject-extracted 'Facebook'): the rejection flips only the key it was attributed to, leaving the other key's copy of the thread 'alive'. Net effect: inflated active-apps badges (including spurious amber >=3 warnings) and undercounted rejections shown to the operator.

### [POSSIBLE (split verdict)] Latest-wins tie-break inverts on equal received_at — second-precision timestamps make ties real

**File:** `apps/web/src/lib/pipeline/bucket.ts`

bucketOutcomes (and the identical latestOf in apps/web/src/lib/applied/unify.ts:110-116) picks the stage-setting row with a strict '>' comparison, so on a received_at tie the FIRST row in served order wins. The server serves rows in received_at ASC order with NO secondary sort key (apps/api/src/job_assist/main.py:3802 — `.order_by(OutcomeEvent.received_at.asc())`; compare other list endpoints which add `.id.asc()`), so among equal timestamps the order is whatever Postgres emits, and the earliest-served (typically the older confirmation) wins — the documented intent at bucket.ts:48-49 is the opposite: 'latest-wins: a rejection after a confirmation moves the card to Rejected'. Ties are not hypothetical: received_at is parsed from the email Date header via parsedate_to_datetime (apps/api/src/job_assist/gmail/client.py:131), which has SECOND precision, and ATS auto-rejections can land in the same second as the confirmation in the same thread. When it happens the card shows 'Still Alive' (or flips nondeterministically between loads) instead of Rejected, on both the Pipeline kanban and the unified Applied/Rejected tabs. The same non-unique ASC sort also makes useAllOutcomes' OFFSET pagination (apps/web/src/lib/api/applied.ts:100-115) able to skip/duplicate a row when equal timestamps straddle a 100-row page boundary — a skipped rejection row silently leaves a card in the wrong column. Fix is a deterministic tiebreak: prefer the terminal/later stage on equal timestamps client-side, and add `.id.asc()` to the /outcomes ORDER BY server-side.

### [CONFIRMED] MAX-watermark + per-message error skip: a transient failure on an older message permanently drops it once a newer message in the same batch succeeds

**File:** `apps/api/src/job_assist/gmail/backfill.py`

Inside _run_email_ingest, a gmail.get_message failure (lines 243-250) or a classifier failure (lines 261-269) just increments a counter and `continue`s — no outcome_event row is written for that message. Gmail's messages.list returns newest-first, so the loop typically inserts the newest message before reaching older ones. If the newer message inserts and an older one then hits a transient error (Gmail 5xx, Gemini 429 exhausted after tenacity's 5 attempts which `reraise=True`s, classifier.py:149-154), the next poll's watermark = the newer message's received_at, and the failed older message falls permanently outside every future `after:` window. Nothing retries it; the module docstring's idempotency claim ('re-running over the same window is safe') only holds for the manual date-range backfill, which the operator has no signal to run — the poll reports success with fetch_errors/classifier_errors buried in counters. Result: silently missing outcomes (e.g. a rejection or interview invite never appears in the Pipeline).

### [POSSIBLE (split verdict)] No session.rollback() in ingest_source's failure handler — a DB-level error (e.g. upsert race IntegrityError) raises PendingRollbackError at commit, loses the failed IngestRun, and aborts the whole fantastic/broad batch

**File:** `apps/api/src/job_assist/services/ingestion.py`

Both upserts are non-atomic SELECT-then-INSERT against real unique constraints (job_posting.py:166 UniqueConstraint('content_hash'), posting_source.py:52 UniqueConstraint('ats','source_job_id')). Two overlapping runs touching the same employer (daily curated cron vs manual /admin/ingest trigger vs broad-ingest — these have separate GH-Actions concurrency groups) can both SELECT none and both INSERT, so one flush raises IntegrityError. The generic except handler (lines 188-196) then mutates `run` and calls `await session.commit()` WITHOUT first rolling back — on a flush-failure the session is in rollback-required state, so commit raises PendingRollbackError: the IngestRun is never marked 'failed' (it was only flushed in the now-dead transaction), contradicting the module docstring's contract 'On exception: mark IngestRun failed' (line 13). The escaping exception then breaks batch isolation: fantastic_ingest's loop has NO per-employer try/except (fantastic_ingest.py:100-141), so its docstring claim 'one bad board never aborts the batch' (lines 84-88) is violated — remaining employers are unswept and ALL last_swept_at stamps are lost (the final commit at line 142 is never reached, falsely tripping the warm_path_fresh health check). broad_ingest.py does catch per-handle (line 334), but then runs count_qualified_broad_this_week on the same poisoned session (line 381), which re-raises PendingRollbackError and aborts the whole sweep, losing the uncommitted lifecycle write-backs.

### [CONFIRMED] Seed backfill_nullables can never fill a NULL tier — excluded on the stale premise that tier is NOT NULL

**File:** `apps/api/src/job_assist/seed.py`

The backfill loop iterates `_SEED_FIELDS - {"name", "tier"}` with the comment "``name`` and ``tier`` are NOT NULL so they're excluded by construction." ### [POSSIBLE (split verdict)] cleanCompany separator split requires whitespace BEFORE the separator — its own documented example fails

**File:** `apps/web/src/lib/pipeline/companyFromSubject.ts`

cleanCompany's tail-cutting split uses /\s+[-–—|:]\s+/, which only matches a separator surrounded by whitespace on BOTH sides. The comment directly above documents '"Acme: Reqs" → "Acme"', but 'Acme: Reqs' has no whitespace before the colon, so the split never fires and the full string survives. Real ATS subjects of the form 'Thank you for applying to Acme: Senior Product Manager' (or 'Acme- Senior PM') therefore produce the company label 'Acme: Senior Product Manager' on Pipeline cards and unified Applied/Rejected rows instead of 'Acme'. Note the sibling roleFromSubject (line 61) deliberately uses \s* on both sides — the asymmetry in cleanCompany contradicts the documented intent. Fix: allow optional leading whitespace for ':'-style separators, e.g. /\s*[-–—|:]\s+/.

### [POSSIBLE (split verdict)] display_name selection is nondeterministic on tied counts (no ORDER BY anywhere in the pipeline)

**File:** `apps/api/src/job_assist/services/company_signals.py`

Neither the outcome query (lines 94-112) nor the contact query (lines 148-154) has an ORDER BY, so Postgres row order is unspecified and can change between requests. display_names are accumulated into a Counter in row order, and `names.most_common(1)[0][0]` breaks count ties by Counter insertion order. So when two raw variants of a key are seen equally often (e.g. 'Stripe' once from a subject and 'Stripe, Inc.' once from a linked row), the display_name returned to the UI can flip between fetches — changing the badge's contacts-link query param (which, combined with the ILIKE substring filter, can flip the click-through between working and empty) and the visible company-pill text. Deterministic tie-breaking (e.g. secondary sort on the name, or ORDER BY received_at) would fix it.

### [POSSIBLE (split verdict)] record_sweep: a hard crash leaves the row 'running' forever (no reaper), and a _finalize failure in the except path replaces the sweep's original exception

**File:** `apps/api/src/job_assist/services/gmail_sweep_run.py`

Two gaps in the autonomous-session pattern. (1) Orphans: the 'running' row commits immediately in its own session (lines 58-60); if the process dies mid-sweep without raising in-process — Railway redeploy, OOM, SIGKILL — nothing ever finalizes it, and no reaper exists anywhere (grep for 'running' shows no cleanup path). The health monitor (main.py:4080-4081, 4112) only checks `status == "failed"`, so a sweep that died 5 minutes in reads as healthy ('running', fresh started_at) for up to 13 hours; given the docstring's stated 5-10 minute backfill duration and routine redeploys, this is a realistic occurrence, though the next cron sweep's row supersedes it. (2) Exception masking: in the failure path (lines 65-67), `await _finalize(...)` performs DB I/O inside the except block before re-raising. If the sweep failed BECAUSE the DB is unreachable, _finalize raises too — the new DB exception propagates instead of the original ('re-raised unchanged' per the docstring is violated), the operator-facing 502 hint from _surface_gmail_failure describes the wrong root cause, and the row additionally sticks at 'running' (case 1).
